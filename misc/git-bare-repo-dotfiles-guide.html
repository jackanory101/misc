Git - Create and add files to repo

Step 1: Initialize the local repo with main branch

  1. Open your terminal and go to the folder you want to track (example: $HOME/tom):
cd $HOME/tom
  1. Initialize a Git repository with main as the default branch:
git init --initial-branch=main

Step 2: Create a .gitignore to control which files to track

  1. Create the .gitignore file in the repo root:
touch .gitignore
  1. Populate it using the “ignore everything except selected files/folders” pattern:
# Ignore everything by default
*

# Whitelist specific files and folders
!.bashrc
!.bash_profile
!.vimrc
!.config/
!.myfolder/**

How it works:

  • * ignores all files and folders by default.
  • !filename un-ignores individual files.
  • !folder/ un-ignores a folder itself.
  • !folder/** recursively un-ignores all files inside a folder.

This way, only the files/folders you want are tracked; everything else is ignored.

So, confusing or what? Apparently this won’t include file:

!.mutt/muttrc

This however will:

!.mutt/
!.mutt/muttrc

However, this will include all files (recursively):

!bash_completion/**

Current cygwin .gitignore file is:

# Ignore everything by default
*

# Except these files/folders
!.mutt/
!.mutt/muttrc
!bash_completion/**
!.bash_aliases
!.bash_functions
!.bash_profile
!.bashrc
!.DIR_COLORS
!.inputrc
!.lynxrc
!.minttyrc
!.vimrc
!tridactylrc
!.gitconfig

Note also !.mutt/* & !.mutt/** - the latter is recursive.

Step 3: Add files to Git

  • Stage all whitelisted files:
git add .
  • You can check what’s staged:
git status
  • Only the whitelisted files/folders should appear under “Changes to be committed”.

Step 4: Commit the files

git commit -m "Initial commit: track selected dotfiles and folders"
  • Your local repository now has a first commit containing only the files you want.

Step 5: Prepare for GitHub

  1. Create a new repository on GitHub (e.g., home-tom).

    • Do not initialize with README, license, or .gitignore.

  2. Add GitHub as the remote in your local repo:

git remote add origin https://github.com/yourusername/home-tom.git
  1. Push your local main branch to GitHub:
git push -u origin main
  • The -u sets the default upstream so future pushes/pulls can be done with just git push or git pull.

Summary

  • .gitignore controls what is tracked — ignore everything by default and whitelist files/folders you want.
  • git add . stages only the whitelisted files/folders.
  • git commit saves the changes in your local repo.
  • Add a GitHub remote and push to sync your local repository.

Once this is set up, you can continue updating your files, staging them, committing, and pushing to GitHub as your home directory changes.

Credential manager

1. What Git Credential Manager is

On Windows, Git can use a credential helper to store and automatically provide your GitHub credentials so you don’t have to type them every time.

  • Credential Manager for Windows (manager-core) is the modern default.
  • It stores credentials securely in the Windows Credential Manager.
  • When you push or pull, Git asks the credential manager for the stored username/password or personal access token (PAT).

So your [credential "helperselector"] selected = manager-core line is telling Git to use this helper.

2. Why you weren’t prompted for credentials

Several possibilities:

  1. Credentials already stored in Windows Credential Manager

    • You may have entered your GitHub credentials or PAT previously.
    • They are cached there, so Git automatically uses them.
    • You can see them in Control Panel → User Accounts → Credential Manager → Windows Credentials → git:

  2. Using SSH instead of HTTPS

    • If your Git remote is an SSH URL (git@github.com:...), and your SSH keys are loaded in ssh-agent, Git pushes without asking for a password.

  3. git-credential-manager-core caching

    • Even for HTTPS remotes, the manager can cache credentials temporarily or permanently depending on setup.
    • That’s why you don’t see a prompt every time.

3. How it works under the hood

  1. You run git push.
  2. Git checks the remote URL (HTTPS or SSH).
  3. Git asks the credential helper (manager-core) for a username/password or token.
  4. Credential Manager returns the stored credentials.
  5. Git completes the push automatically.

No manual input is needed if credentials are already stored.

Summary

  • Git Credential Manager automates authentication with remotes.
  • It can store credentials in Windows Credential Manager, cache them, or manage SSH keys.
  • That’s why you could push to GitHub without being prompted — your credentials were already stored or cached.

Remove file(s) from cache

Here’s a clean way to stop tracking multiple files in Git while keeping them on disk, plus a quick explanation of the underlying issue.

1. One-liner to stop tracking multiple files

If you have several files you accidentally added (e.g., viminfo, *.log, temporary configs), you can run:

git rm --cached file1 file2 file3

Or, if you want to remove all currently tracked files that are now in .gitignore:

git rm -r --cached .
git add .
git commit -m "Stop tracking files now in .gitignore"

Explanation:

  • git rm -r --cached . → removes all files from the index (staging area), but leaves them on disk.
  • git add . → re-adds only files not ignored (thanks to your .gitignore).
  • Commit → finalizes the change, so Git stops tracking ignored files.

2. Quick summary of the issue

  1. Git tracks files, not ignore rules.
  2. .gitignore only affects untracked files.
  3. Once a file is staged or committed, it continues to be tracked even if you remove it from .gitignore.
  4. To stop tracking: use git rm --cached <file> and commit.

Key takeaway:

  • .gitignore prevents new files from being added accidentally.
  • For already-tracked files, you need to remove them from Git’s index to stop tracking.

File(s) still on remote

so i rm file from cache. and local git is ignoring it. however, remote repo still has the file

Exactly — that’s expected behavior. Let me explain step by step:

1. What you did locally

git rm --cached viminfo
  • Removes viminfo from the index (staging area).
  • Leaves the file on your disk.
  • Because .gitignore no longer whitelists it, Git now ignores it locally.

2. Why the file is still on the remote

  • Git only removes files from the remote when you commit the removal and push.
  • Right now, the removal is staged locally but hasn’t been committed.

3. How to remove it from the remote

  1. Commit the change:
git commit -m "Stop tracking viminfo"
  1. Push to remote:
git push origin main
  • This tells GitHub (or whatever remote) to remove the file from the repository.
  • The file stays on your local disk, but it will no longer exist in the remote repo.

Key points

Action Local Remote
git rm --cached file File removed from index, still on disk Still exists
git commit Confirms removal in local repo Not yet reflected
git push Sends removal to remote File removed from remote

So in short: after rm --cached, always commit and push to remove the file from the remote repository.

About Git

1. Git is a version control system

  • Git tracks changes to files over time.
  • You can review history, revert changes, collaborate with others, and maintain multiple lines of development (branches).
  • Unlike some version control systems, Git is distributed: every clone has the full repository, including history.

2. The three main “areas” in Git

Git manages files through three primary states:

State Description
Working Directory Your files on disk that you can edit freely.
Staging Area (Index) Files you have marked to include in the next commit. Think of it as a “draft commit.”
Repository (Local Git Database) Where committed snapshots live. Each commit is a complete record of the tracked files at a point in time.

Why stage before committing?

  • Staging allows you to select exactly which changes go into a commit, instead of committing everything.
  • Example: You edit .bashrc and .vimrc but only want to commit .vimrc. Staging .vimrc alone lets you commit it without touching .bashrc.

Commands:

git add <file>     # stage changes
git commit -m "message"  # save staged changes as a commit

3. Commits

  • A commit is a snapshot of your staged files.
  • Each commit has a unique SHA hash, author info, timestamp, and a commit message.
  • Commits form a history of your project.

Good practice: Make commits small and logical, with descriptive messages.

4. Branches

  • A branch is a movable pointer to a commit.
  • Default branch is usually main.
  • Branches allow parallel development:
main → stable code
feature-x → work on a new feature
  • You can switch between branches, merge changes, and experiment without affecting the main line of development.

Commands:

git branch <name>       # create a branch
git checkout <branch>   # switch to branch
git switch <branch>     # modern alternative to checkout
git merge <branch>      # combine changes from another branch

5. Remote Repositories

  • Git supports remote repositories (e.g., GitHub, GitLab).
  • Push local commits to share them, pull changes to update your local copy.

Commands:

git remote add origin <url>
git push -u origin main
git pull
  • -u sets upstream, making future git push and git pull easier.

6. Ignoring files

  • .gitignore prevents Git from tracking certain files (e.g., temp files, compiled binaries, personal configs).
  • Only untracked files are affected; tracked files remain until removed with git rm --cached.

7. Undoing changes

  • Unstage a file: git reset <file>
  • Discard local changes: git checkout -- <file>
  • Stop tracking a file but keep locally: git rm --cached <file>

8. Summary of workflow

  1. Edit files in your working directory.
  2. Stage changes you want to include: git add <file>
  3. Commit staged changes: git commit -m "message"
  4. Optionally, push commits to a remote: git push
  5. Use branches to experiment or develop features separately.
  6. Merge branches back into main when ready.

9. Key concepts to remember

  • Git tracks snapshots, not differences (each commit is a full picture).
  • Staging lets you control what goes into a commit.
  • Branches allow parallel work without interfering with main history.
  • .gitignore protects against committing files you don’t want in version control.
  • Remote repositories let you share and sync changes with others.

Practical example of branches use

Scenario: Managing dotfiles

You have a main branch with your stable home configuration files tracked. You want to try a new Vim plugin without risking your current setup.

Step 1: Create a branch for experimentation

git branch vim-experiment   # create a new branch
git switch vim-experiment   # switch to it
  • Now any changes you make are isolated from main.
  • You can safely install the plugin, update .vimrc, or add new files.

Step 2: Work on the branch

# Edit .vimrc and add plugin config
vim ~/.vimrc
  • Stage and commit your changes:
git add .vimrc
git commit -m "Test new Vim plugin configuration"
  • These commits exist only in vim-experiment.

Step 3: Decide whether to keep changes

  1. If it works and you like it:

    • Merge back into main:
    git switch main
git merge vim-experiment
    • Now main includes the plugin changes.

  2. If it doesn’t work:

    • Simply delete the branch:
    git branch -d vim-experiment
    • Your main branch remains untouched, no messy configs or broken setups.

Why branches are useful

Benefit Explanation
Isolation Test new features without affecting main work
Experiment safely Break things on a branch, then discard if needed
Parallel work Work on multiple features at the same time
Clean history Merge only completed, tested changes into main

Optional: Push branches to remote

git push -u origin vim-experiment
  • Lets you backup the branch or collaborate with others without touching main.

Key takeaway: Branches let you experiment, develop, or fix things safely, keeping the main line stable. They’re one of Git’s most powerful features for both personal projects and collaborative work.

Discard changes and go back to original file

Git lets you discard changes in your working directory and go back to the state of a file from the last commit. Here’s how to handle it depending on your situation:

1. Discard changes to a single file

If you’ve modified a file but haven’t staged it yet:

git checkout -- filename

or (modern syntax):

git restore filename
  • Replaces the file with the version from the current commit (HEAD).
  • Local changes will be lost, so make sure you really want to discard them.

2. Discard changes in multiple files

git restore file1 file2

or

git restore .
  • The . restores all files in the working directory to the last commit.

3. If changes are already staged

If you used git add but now want to undo that:

git reset HEAD filename
  • Unstages the file but keeps the changes on disk.
  • Then you can run git restore filename if you want to fully revert it.

4. Restore to an older commit

If you want to go back to a file as it was in a specific past commit:

git checkout <commit-hash> -- filename

or

git restore --source <commit-hash> filename
  • This pulls the file from that commit into your working directory without changing the branch pointer.

Quick summary table

Scenario Command
Undo unstaged changes in a file git restore filename
Undo all unstaged changes git restore .
Unstage a file git reset HEAD filename
Restore a file from a previous commit git restore --source <commit> filename

Key takeaway: Git keeps the last committed version safe, so you can experiment freely. Use git restore (or git checkout --) to discard unwanted changes and revert a file to a known good state.

Step-by-step example

Here’s a step-by-step workflow for experimenting on dotfiles using Git branches, while keeping the ability to roll back individual files safely.

Step 1: Create a branch for experimentation

git branch experiment
git switch experiment
  • Your main branch remains stable.
  • All experimental changes happen on experiment.

Step 2: Edit files freely

  • Open or modify any config files, e.g., .vimrc:
vim ~/.vimrc
  • Git will detect these as modified files in your working directory.

Step 3: Check which files have changed

git status
  • Shows modified files not yet staged.
git diff
  • Shows exactly what changed line by line.

Step 4: Stage and commit if you like changes

git add ~/.vimrc
git commit -m "Test new Vim plugin"
  • Commit only the files you want to save.
  • Others remain uncommitted, so you can still discard them.

Step 5: Roll back unwanted changes

A. Unstaged changes

If you edited a file but don’t want to keep the changes:

git restore ~/.vimrc
  • Restores the file to the state in the last commit on this branch.
  • Other files are unaffected.

B. Staged changes

If you already ran git add but want to undo:

git reset HEAD ~/.vimrc   # unstage
git restore ~/.vimrc      # discard changes
  • This ensures the file is no longer staged and reverts it fully.

C. Roll back to an older commit

If you want a file as it was in a previous commit:

git restore --source <commit-hash> ~/.vimrc
  • Only changes that file; your branch pointer does not move.

Step 6: Merge successful experiments

  • Once you’re happy with some changes:
git switch main
git merge experiment
  • Only committed changes on experiment are merged into main.
  • Any uncommitted or discarded experiments remain safely on the experiment branch.

Step 7: Clean up experiment branch

  • If you’re done and everything on experiment is either merged or discarded:
git branch -d experiment
  • Deletes the branch safely.

Key Advantages

  • Isolation: You can experiment without touching main.
  • Selective rollback: Restore individual files without affecting others.
  • Commit flexibility: Save only the experiments you like; discard the rest.
  • Safe cleanup: Merge or delete branches after testing.

Further notes

1. Your repo is a snapshot

  • When you first commit your Vim files, Git stores a snapshot of all tracked files at that moment.
  • This snapshot is identified by the commit hash.
git log --oneline
  • The first commit is your “original snapshot.”
  • Every commit after that is a new snapshot, reflecting the changes staged at that time.

2. Rolling back to the original snapshot

You have a few options depending on what you want:

A. Restore a single file

git restore --source <commit-hash> ~/.vimrc
  • Replaces ~/.vimrc with the version from the original commit.
  • Other files remain untouched.

B. Restore all files to that commit (working directory only)

git restore --source <commit-hash> .
  • Your working directory will reflect the original snapshot.
  • No commits are changed; it’s just a local rollback.

C. Reset the branch to that commit (destructive)

git reset --hard <commit-hash>
  • Moves the branch pointer back to the original snapshot.
  • All commits after that are removed from this branch.
  • Be careful — this can permanently discard later commits if not backed up.

3. Viewing the original snapshot

  • You can always see the files in a commit without changing your working directory:
git show <commit-hash>:~/.vimrc
  • Or browse the commit in a Git GUI or GitHub.

4. Key points

  • Yes, you can always roll back to the original snapshot, because Git stores the full history.
  • You can restore individual files or the entire repo.
  • Rolling back doesn’t affect GitHub unless you push changes that modify history.
